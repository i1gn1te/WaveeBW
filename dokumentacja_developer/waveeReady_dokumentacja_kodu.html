waveeReady - Bardzo Szczegolowa Dokumentacja Kodu

Dokumentacja opisuje pliki po kolei: najpierw backend, potem frontend. Jezyk jest celowo prosty, z naciskiem na to co robi plik, jak dziala i co jest trudniejsze do zrozumienia.

Ta dokumentacja dotyczy plikow projektowych aplikacji. Nie opisuje setek plikow z node_modules, bo to biblioteki zewnetrzne.

Spis tresci

1. Jak dziala caly projekt 2. Backend - plik po pliku 3. Frontend - plik po pliku 4. Najtrudniejsze mechanizmy

1. Jak dziala caly projekt

waveeReady to aplikacja webowa oparta na modelu:

  • Frontend (React + Vite + Tailwind) - pokazuje interfejs, pobiera dane z API, obsluguje logowanie i nawigacje.
  • Backend (Express + Prisma) - udostepnia endpointy REST, komunikuje sie ze Spotify API i baza danych.
  • Baza danych (Prisma schema + PostgreSQL) - zapisuje uzytkownikow, recenzje, playlisty i utwory polubione.

Przeplyw: uzytkownik loguje sie (Spotify albo demo) - frontend dostaje sesje przez cookie - frontend wywoluje API - backend sprawdza token - backend czyta/zapisuje dane i ewentualnie pyta Spotify.

2. Backend - plik po pliku

backend/package.json

Rola: konfiguracja projektu backendowego Node.js.

  • Definiuje skrypty: uruchamianie serwera, tryb dev, komendy Prisma.
  • Trzyma zaleznosci: Express, Prisma, axios, JWT, Zod, CORS, cookie-parser.
  • Ustawia type: commonjs (moduly CJS).
backend/prisma/schema.prisma

Rola: model danych aplikacji.

  • User: profil uzytkownika i ulubione gatunki.
  • Review: recenzja utworu (ocena + tekst + cechy audio).
  • Playlist i PlaylistTrack: playlisty lokalne i utwory na listach.
  • LikedTrack: polubione utwory z dodatkowymi cechami.
  • Kluczowe ograniczenia: unikalnosc recenzji (userId + trackId), unikalnosc utworu w playliscie.
backend/src/index.js

Rola: punkt startowy API.

  • Tworzy aplikacje Express.
  • Konfiguruje CORS (localhost + siec lokalna).
  • Wlacza JSON parser i cookie-parser.
  • Podpina trasy: auth, user, reviews, spotify, playlists, recommendations.
  • Dodaje health-check i globalny error handler.
backend/src/lib/prisma.js

Rola: singleton klienta Prisma.

  • Tworzy jedno polaczenie klienta Prisma.
  • W dev zapisuje klienta w globalThis, zeby uniknac wielu instancji po hot-reload.
backend/src/lib/mockData.js

Rola: dane testowe dla trybu demo bez Spotify.

  • Tablice mock utworow i artystow (z obrazkami albumow).
  • Mapa mock cech audio (tempo, energia, valence itd.).
  • Lista gatunkow oraz funkcja zamiany numeru tonacji na nazwe.

To dzieki temu aplikacja dziala nawet, gdy Spotify API nie jest skonfigurowane.

backend/src/lib/routeHelpers.js

Rola: wspolne helpery dla tras.

  • handleSpotifyRouteError: mapuje bledy Axios/Spotify na sensowne odpowiedzi HTTP.
  • isDemoUser: rozpoznaje token demo.
  • shuffleArray: losowe mieszanie tablicy.
backend/src/lib/spotify.js

Rola: warstwa integracji ze Spotify API.

  • OAuth: generowanie URL logowania, wymiana code - token, odswiezanie tokenu.
  • Pobieranie danych: profil usera, top tracks/artists, szukanie utworow, dane utworu, audio features.
  • Rekomendacje bez starego endpointu: "podobne" przez related artists i ich top tracks.
  • Discovery po gatunku: wyszukiwanie zapytaniem "genre:...".
  • Playlisty: tworzenie playlisty i dodawanie utworow po URI.

Wazne: audio-features endpoint Spotify bywa ograniczony. Kod obsluguje to i zwraca null zamiast twardego bledu.

backend/src/middleware/auth.js

Rola: ochrona endpointow i automatyczne odswiezanie sesji.

  • Sprawdza token JWT z cookie albo naglowka Authorization.
  • Po poprawnej walidacji ustawia req.userId i req.spotifyAccessToken.
  • Jesli token wygasl, probuje odswiezyc access token przez spotifyRefreshToken.
  • Po odswiezeniu ustawia nowe cookie i przepuszcza dalej.
backend/src/routes/auth.js

Rola: caly system logowania i sesji.

  • /register i /local-login: lokalne konto email + haslo (hash scrypt + salt).
  • /login: zwraca URL do logowania Spotify.
  • /demo-login: szybkie logowanie testowe.
  • /callback: obsluga OAuth callback, pobranie profilu Spotify, upsert usera, JWT cookie, redirect do dashboardu.
  • /me, /refresh, /logout: zarzadzanie sesja i profilem.

Kod callback ma dodatkowa ochrone przed ponownym uzyciem tego samego code (Set processedCodes).

backend/src/routes/user.js

Rola: profil i statystyki uzytkownikow.

  • Pobieranie i edycja profilu (bio, displayName, favoriteGenres).
  • Statystyki recenzji (srednia, rozklad ocen).
  • Wyszukiwarka innych userow (bez demo_user).
  • Publiczny profil innego usera po userId.
backend/src/routes/reviews.js

Rola: CRUD recenzji i feed recenzji.

  • POST /: tworzy lub nadpisuje recenzje tego samego utworu przez upsert.
  • W trybie Spotify pobiera dane utworu i cechy audio; w demo bierze mocki.
  • /my, /track/:trackId, /:reviewId, DELETE, oraz feed globalny.
  • Jest paginacja i podstawowe sortowanie.
backend/src/routes/spotify.js

Rola: endpointy stricte "muzyczne" pod frontend.

  • Wyszukiwarka utworow z limitem i fallbackiem demo.
  • Szczegoly utworu + audio features.
  • Top tracks, top artists i lista gatunkow.
  • Clamp limitu, zeby unikac skrajnych wartosci.
backend/src/routes/playlists.js

Rola: lokalne playlisty i synchronizacja do Spotify.

  • Tworzenie playlisty, pobieranie swoich i pojedynczej playlisty.
  • Dodawanie/usuwanie utworow z kontrola wlasciciela.
  • /sync: tworzy playliste na Spotify i wrzuca track URI.
  • Usuwanie playlisty.
backend/src/routes/recommendations.js

Rola: rekomendacje i odkrywanie muzyki.

  • /similar/:trackId: utwory podobne do podanego tracka.
  • /discover/:genre: odkrywanie po gatunku.
  • W demo zwraca losowo wymieszane mockTracks.

4. Najtrudniejsze mechanizmy - proste wyjasnienie

A) Tokeny i sesja (backend + frontend)

  • Backend zapisuje JWT w cookie po logowaniu.
  • Middleware sprawdza token dla chronionych tras.
  • Jesli access token wygasl, middleware lub interceptor frontendu uruchamia odswiezanie.
  • Dzieki temu user czesto nawet nie widzi, ze sesja byla odnawiana.

B) Dlaczego sa tryby Spotify i Demo

  • Gdy Spotify API nie jest skonfigurowane, app dalej ma dzialac.
  • Wtedy backend zwraca mockowe utwory i cechy audio.

C) Upsert recenzji

  • Nie ma duplikatow recenzji tego samego utworu przez tego samego usera.
  • Kolejne zapisanie recenzji tego tracka po prostu aktualizuje poprzednia.

D) Rekomendacje po zmianach Spotify API

  • Stary endpoint recommendations zostal ograniczony/usuniety dla czesci zastosowan.
  • Projekt obchodzi to przez: podobni artysci -> ich top utwory -> mieszanie i deduplikacja.

E) Dlaczego w kodzie jest duzo "invalidates" i "clear cache"

  • React Query trzyma cache danych.
  • Po mutacji (np. dodanie recenzji) trzeba odswiezyc konkretne query, zeby UI pokazal aktualny stan.

Podsumowanie dla Ciebie (jak odpowiadac na pytania)

  • Jesli pytanie dotyczy "gdzie jest logowanie" -> auth route + auth middleware + AuthContext + Login/Home.
  • Jesli pytanie dotyczy "skad sa rekomendacje" -> recommendations route + spotify lib (findSimilarTracks / discoverByGenre).
  • Jesli pytanie dotyczy "gdzie sa recenzje" -> reviews route + Track page + Reviews page + schema model Review.
  • Jesli pytanie dotyczy "playlista" -> playlists route + PlaylistGenerator page.
  • Jesli pytanie dotyczy "czemu cos nie dziala" -> najpierw api.ts (frontend), potem odpowiadajacy route w backendzie.

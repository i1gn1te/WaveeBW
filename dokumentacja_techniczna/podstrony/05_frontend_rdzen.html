05. Frontend rdzen i komunikacja

To jest serce frontendu: start appki, routing, sesja i API client.

Powrot do strony glownej

1) frontend/src/main.tsx

Tu rusza React i konfiguracja cache query. QueryClient trzyma dane z API i ogranicza liczbe zapytan do backendu. 

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60 * 5,
      retry: 1,
    },
  },
})

2) frontend/src/App.tsx

Tu jest mapa wszystkich tras. Czesci prywatne sa opakowane przez ProtectedRoute.

  • Publiczne: Home, Login
  • Chronione: Dashboard, Search, Track, Reviews, Generator, Community, Profile

3) frontend/src/contexts/AuthContext.tsx

Context daje globalny stan zalogowania i usera. Przy starcie probuje pobrac /auth/me. 

const { data } = await getCurrentUser()
setUser(data)

Przy logout czyści cache query przez queryClient.clear(), wiec nie ma starych danych po sesji.

4) frontend/src/lib/api.ts

To jeden wspolny klient HTTP. Najwazniejszy jest interceptor 401, ktory sam probuje odswiezyc sesje. 

if (!originalRequest || status !== 401 || originalRequest._retry || shouldSkipRefresh) {
  return Promise.reject(error)
}
originalRequest._retry = true
await api.post('/auth/refresh')
return api(originalRequest)

Dlaczego to jest dobre

  • Nie trzeba recznie obslugiwac refresh w kazdej stronie.
  • User rzadziej widzi logowanie od nowa.
  • Logika API jest skupiona w jednym miejscu.

5) Layout i ProtectedRoute

  • Layout.tsx daje wspolna nawigacje i stopke.
  • ProtectedRoute.tsx blokuje wejscie bez sesji.

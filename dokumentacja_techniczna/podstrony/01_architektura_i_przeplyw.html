01. Architektura i przeplyw danych

Tu jest rozpisane krok po kroku jak request idzie przez caly system.

Powrot do strony glownej

1) Co to jest za typ aplikacji

waveeProjectBW to klasyczny uklad klient-serwer. Klientem jest frontend React, serwerem backend Express. Dodatkowo backend gada ze Spotify API i z baza PostgreSQL przez Prisma.

  • Frontend odpowiada za UI i interakcje usera.
  • Backend odpowiada za auth, logike biznesowa i zapis danych.
  • Baza przechowuje userow, recenzje, playlisty i inne dane trwale.

2) Przeplyw requestu - prosty scenariusz

  1. User otwiera strone i np. szuka piosenki.
  2. Frontend wysyla GET na /api/spotify/search?q=....
  3. Backend sprawdza autoryzacje przez middleware.
  4. Backend pobiera dane ze Spotify (lub mockData w demo).
  5. Backend zwraca JSON.
  6. Frontend renderuje wynik jako karty utworow.

3) Przeplyw auth (normalny i demo)

Wariant A: Spotify OAuth

  1. Frontend odpytuje /api/auth/login po URL logowania.
  2. Backend zwraca URL do Spotify accounts.
  3. User loguje sie w Spotify i wraca na /api/auth/callback.
  4. Backend wymienia code na token i robi JWT cookie.

Wariant B: Demo

  1. Frontend wywoluje /api/auth/demo-login.
  2. Backend tworzy/odswieza usera demo i ustawia cookie.
  3. Aplikacja dziala na mockData bez prawdziwych tokenow Spotify.

4) Najwazniejsze punkty architektury
